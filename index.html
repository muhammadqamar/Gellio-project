About the Galileo Platform
Galileo’s APIs connect you to the most advanced capabilities in fintech and payments. The Galileo Platform enables our Partners to perform transactions between any parties. Our Partners’ transactions typically include financial transactions such as credit/debit/ATM transactions, transfers between accounts, and disbursement of funds. We connect to the various association networks (i.e. Visa, Mastercard, Cirrus, Maestro) on behalf of our Partners. Integrate with the Galileo Platform through our web services APIs, batch file processing, and hosted solutions. We provide a flexible framework for enabling the creation of accounts and facilitating payments.

Galileo product hierarchy
The Galileo platform is organized in a hierarchy as shown below -
Partner
The Galileo Platform is organized in a hierarchy. At the highest level is the Partner, or a Galileo customer. Each Partner can have one or more programs.
Program
A wide variety of programs can be created with the Galileo Platform. Our Partners create credit, debit, and prepaid cards programs with unique and customized features.
Product
A program will have one or more products. Products allow Partners to provide options and offerings to their account holders. Product level system and business configurations provide the functionality of the program. Examples of these configurations include account type, account portal capability, account limitations, card design, account messaging and alerting, and so on.
Account
Each account belongs to a product. The assignment of the product can change for an account. Hundreds of product configurations can be set-up on the Galileo platform.
Account Structure
The Galileo account structure is unique and flexible. The account structure:
  • Account holder
  • Account
  • Balance
Each account is based on an account number (PRN) and a product ID. (A card can be associated with an account but is not required). The PRN is a unique number that allows identification of the account for many purposes, such as API calls.
An account holder can have multiple accounts, any of those accounts can share a balance with other accounts and account holders.
Account examples:
  • Basic account
  • Secondary account holders
  • Secondary account holders with an exclusive balance
  • Savings accounts
The Galileo Platform account structure is built on an account holder, an account, and balance structures. Accounts usually belong to individuals or entities. Accounts must have an identifier (i.e. Account number) and a balance. Other attributes of an account
  • Personal identifiable information
  • Account type
  • Account access methods (i.e. credit card numbers)
Secondary accounts
Multiple accounts share a balance, and such accounts may have same or different account holders. All transactions for those accounts will affect the same balance. If there are insufficient funds in the account that is being transacted against, that transaction will be declined.
Secondary account holders with an exclusive balance
Each account maintains exclusive balance and any transactions affects this balance only.
Savings account
This is a basic account structure on the Galileo platform. It will have an exclusive balance. There will not be a card associated with the savings account, and the only means by which an account holder can access the funds is via an account to account transfer.
There are no technical limitations to the number of accounts an account holder can have. There are no technical limitations to the number of accounts that can share the same balance.
Note
A secondary account set to share a balance cannot have an exclusive balance. Likewise, an account set to have an exclusive balance cannot share an existing balance.

Getting Started with the Program Master API
Using the Galileo Platform and the Master Program API requires a sign up through Galileo’s account services. See the steps below to get started.
Signing up
Become a Galileo Partner by contacting us with the registration form found here. (But first, we recommend you give our Sandbox a try.)
Using the Program Master API
Once a Partner has completed registration with Galileo (using the registration form mentioned above) the steps to get up and running with live calls to the API are straightforward.
Using our Getting Started form, a Partner: Creates API keys, creates a product, creates a team, make calls to the API, and uses the returned information as the Partner sees fit. Steps in using the Getting Started form:
  1. Create confidential keys to use with the API.
  2. Secure the confidential keys, and share them with the Partner’s development team.
  3. Create a product. (Steps for this are found in the documentation).
  4. Fund an account.
  5. Create a development team.
  6. Make API calls and consume the returned information.
Keys for using the API are shown below. (You get these as part of the registration process).
  • apiLogin - Provider's web service username.
  • apiTransKey - Provider's web service password.
  • providerId - Galileo issued provider identifier.
  • prodId - The identity of the account or card account being created.
  • transactionId - A string of alphanumeric characters (hyphen also allowed) that helps uniquely define your transaction on the Galileo Platform.
Your account manager will work with you to be sure you have the correct information.
Sample API call and response
A sample API call using cURL with Sandbox keys looks like the below. (The ping() method is demonstrated).
Note the endpoint:
curl -v https://sandbox-api.gpsrv.com/intserv/4.0/ping
Note also the keys being used:
  • apiLogin
  • apiTransKey
  • providerId
  • transactionId
  • prodId
The ping() call is very simple:
# The following shell script will use cURL to call ping() and
# print the full XML response
curl -v https://sandbox-api.gpsrv.com/intserv/4.0/ping \
-d 'apiLogin=Cl6NYX-9999' \
-d 'apiTransKey=WVKfS3eQu5' \
-d 'providerId=550' \
-d 'transactionId=54352-random-string-68775' \
-d 'prodId=501'
A XML response to the ping() method call:

< ?xml version="1.0" encoding="UTF-8">
   <response>
   <status_code>0</status_code>
   <status>Success</status>
   <system_timestamp>2011-10-07 10:25:07</system_timestamp>
   <response_data>
       <processing_time>0.973</processing_time>
   <echo>
       <transaction_id>12345a</transaction_id>
       <provider_transaction_id>77bb</provider_transaction_id>
       <provider_timestamp>2013-02-06 10:10:10</provider_timestamp>
   </echo>
</response>

Note that the status_code statement returned a 0, which means a successful ping() transaction.
Understanding the endpoint and the response file makes it easy to use them in developing applications.

Using the Sandbox API
The Galileo Processing Sandbox is a trial version of our solution, or a look into the possibilities. Using the Sandbox API builds expertise in the live Program Master API. The Sandbox uses simulated data to create simple card programs and products.
Create a Sandbox account from the Galileo sign up page. Enter a few fields of contact information, and we’ll be in touch to complete the registration.
Once registered, these are the steps in using the Sandbox:
  1. Get registered with Galileo for Sandbox access.
  2. Set up a programming environment. (The Sandbox supports cURL, Java, Python, PHP, C#, and Ruby).
  3. Using the keys supplied from Sandbox registration, send a ping() call to the Sandbox API.
  4. Examine the response (the response will be in JSON or XML). A status_code = 0 in the response file means a successful call.
  5. Advanced class: send a createAccount() call with parameters to the Sandbox.
  6. Experiment -- Create programs and products. Everything a developer learns will apply to the Program Master API. Our documentation contains guides and references to using the API methods.
See Using the Program Master API for a sample ping() call, and the response file it generates. The next section contains information on the createAccount() method call response.

Transactions
A financial transaction is a change in the status of the finances of two or more businesses or individuals. In credit and debit card processing, transactions can post to an account or they can pend. Use the Galileo Platform to execute transactions that:
  • Load funds to a customer account.
  • Assess fees to an account.
  • Reverse a fee created with the assessFee() method or created elsewhere on the Galileo platform.
  • Makes credit or debit adjustments on an account.
  • Reverse a credit or debit adjustment on an account.
  • Reverse a payment.
  • Report pending ACH deposits (including status information).
  • Create a hold or reserve on an amount of funds for a specific payment and a given amount of time.
Many other transaction types are available.
Methods
The Galileo Platform uses the Master Program API to execute method-based transactions. Methods pass a method name and parameters over a secure URL. A response with the desired information is returned in JSON or XML format. A partial list of methods that execute transactions.
  • assessFee()--Assesses a fee to an account.
  • reverseFee()--Reverse a fee created with the assessFee method or created elsewhere on the Galileo platform.
  • createAdjustment()--Makes credit or debit adjustments on an account.
  • reverseAdjustment()--Reverses a credit or debit adjustments on an account.
  • createPayment()--Enables the loading of funds to a customer account.
  • reversePayment()--Reverses a payment.
  • updatePayment()--Allows for the updating of the 'hold days' for a pending payment.
  • createAccountTransfer()--Bundles a load, adjustment, and fee into one API call. Transfers occur between two accounts on a single program.
  • reverseAccountTransfer()--Reverses a createAccountTransfer() method call.
  • expireAuthorization()--Expires an existing authorization hold on a customer account.
  • getPendingDeposits()--Get a report of pending ACH deposits including status information.
  • getDepositHistory()--Get deposit history data for your entire program or related to a specific customer.
  • modifyPendingDepositStatus()--Triage direct deposits; accepting and rejecting as needed
  • createHold()--Creates a hold or reserve on an amount of funds for a specific payment and a given amount of time.
  • expireHold()--Expires a funds hold made with createHold().
  • getHoldHistory()--Returnsa history of holds created for a specified account.
  • getPendingMerchantCredits()--Returns a data set of merchant credits that are pending or in a status of waiting to be processed.
  • updatePendingMerchantCredit()--Post or post/hold a merchant credit.

Security
Each method uses a base set of parameters that handle validation of the authentication elements. The user is authenticated before the method logic is processed.

Locations in the Galileo Platform have multiple uses. Examples include:
Configurations where locations are used for instant issue card management and configurations where locations are used to relate and identify where a cardholder was issued their card. The following methods can be used to manage locations.
Locations
Locations are used to identify where a cardholder was issued their card (and to store configurations for instant issue card management). Some card associations require location information. The Master Program API can:
  • Returns a list of load locations nearest the postal code that is provided.
  • Retrievesa recordset of locations based on search criteria. Locations can be searched by location ID, location name (or partial), and location street address (or partial).
  • Creates a new store location, associates a provider specific ID, and nests it below a parent store.
  • Updates or modifies key data elements related to a location.
The group_id response field (available to several methods) reports the location (store, entity, etc.) of a customer acquisition. The group_idresponse is set using the location and locationType parameters in location based methods.

The modifyPurchasingCard() method is used to retrieve the current configuration for a purchasing card account. To get the current configuration without any modifications, call the method without passing newProdId, newStoreId, or newApprovedMerchantAuthIds values.
Modify a purchasing card
The modifyPurchasingCard() method allows the Partner to perform the following functions on a card account designated for purchasing:
  • Change the product ID.
  • Change the store ID the purchasing card account is assigned to (for tracking purposes). See the 'Location Management' section of this document for help creating locations or getting a current recordset of locations.
  • • Modify what merchants the purchasing card can be used with. A CSV list of authorization IDs is passed as the 'newApprovedMerchantAuthIds' parameter value. The merchant ID / merchant description and acquirer ID are retrieved from those example authorizations and used as examples. The card account is updated to only allow transactions from those merchants.
The modifyPurchasingCard() method can also be used to retrieve the current configuration for a purchasing card account. To get the current configuration without any modifications, call the method without passing newProdId, newStoreId, or newApprovedMerchantAuthIds values.
